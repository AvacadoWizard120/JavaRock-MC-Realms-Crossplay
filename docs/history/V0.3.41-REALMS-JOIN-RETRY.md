# v0.3.41 Realms join retry

This patch handles transient Realms API failures during bridge startup.

## What changed

- Realm join endpoint lookup now retries transient `408/425/429/500/502/503/504` failures before giving up.
- `REALM_JOIN_MAX_ATTEMPTS=0` now means keep retrying until the endpoint recovers or the process is stopped.
- Retry timing can be tuned with:
  - `REALM_JOIN_MAX_ATTEMPTS`
  - `REALM_JOIN_RETRY_BASE_MS`
  - `REALM_JOIN_RETRY_MAX_MS`
  - `REALM_JOIN_RETRY_JITTER_MS`
- `run-bridge-via-bedrock-relay-latest.ps1` and `run-bedrock-packet-recorder-latest.ps1` default to patient unbounded startup retries with slow backoff.
- ViaBedrock downstream endpoint refreshes use their own bounded retry defaults, then fall back to the already-selected endpoint:
  - `NETHERNET_RELAY_REFRESH_REALM_JOIN_MAX_ATTEMPTS`
  - `NETHERNET_RELAY_REFRESH_REALM_JOIN_RETRY_BASE_MS`
  - `NETHERNET_RELAY_REFRESH_REALM_JOIN_RETRY_MAX_MS`
  - `NETHERNET_RELAY_REFRESH_REALM_JOIN_RETRY_JITTER_MS`
- ViaBedrock relay mode no longer starts the Java status fallback when the selected NetherNet endpoint cannot be resolved.

## Why

The bridge could list Realms successfully but fail on `/worlds/<id>/join` with:

```text
503 Service Unavailable Retry again later
```

Before this patch, that failure fell through into a status-only Java LAN server, which made the Java client look connected while the Realm bridge was not actually running.
