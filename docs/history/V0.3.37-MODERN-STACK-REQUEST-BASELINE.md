# v0.3.37 Modern Stack Request Baseline

This patch uses the native Bedrock recorder output to close the largest remaining crafting/inventory protocol gap.

## Findings

Native Bedrock 1.26.20 uses `item_stack_request` for inventory movement and player 2x2 crafting. The baseline run showed:

```text
hotbar -> cursor:                     take
cursor -> 2x2 crafting input slot 29:  place
craft output:                         craft_recipe, results_deprecated, consume, take
cursor -> hotbar/inventory:            place
```

The Java bridge path still emitted legacy normal `inventory_transaction` packets after the patched ViaBedrock local UI model updated the Java client. That explains why crafting could look locally plausible but still remain fragile against Realm authority.

## Changes

- `item_stack_response` normalization now keeps explicit empty `custom_name` and `filtered_custom_name` fields. v0.3.36 dropped responses because the local ViaBedrock schema tried to serialize an undefined string.
- The live 2x2 recipe DB now preserves Bedrock `recipe.network_id`.
- The Node relay rewrites safe legacy normal inventory transactions into modern `item_stack_request` packets:
  - player/hotbar/inventory/crafting-input move commits become `take` + `place`;
  - 2x2 craft commits become `craft_recipe` + `results_deprecated` + `consume` + `take`, followed by `place`.
- The rewrite is limited to safe player inventory / hotbar / player 2x2 crafting slots.

## Still not final

This does not implement crafting-table/workbench yet. It also does not rewrite chest/container item-stack moves yet. That needs a native chest baseline because open containers use different Bedrock slot descriptors.
