# V0.3.56 - Authoritative Cursor Resync

## Why this patch exists

The 2026-06-12 run showed regular pickup/place could succeed, but the Java shim later reopened or closed inventory while still believing the cursor held an old item stack. The next click then emitted a valid-looking Bedrock `item_stack_request` from `CursorContainer[0]`, but the Realm rejected it because the server-side cursor was already empty.

The visible symptom was item stacks working for a moment, then teleporting back or evaporating after an inventory close, a shift-click, or another delayed authoritative inventory update.

## What changed

- `InventoryContainer` now resyncs its private carried cursor from ViaBedrock's HUD cursor slot before player-inventory clicks.
- Player inventory close also resyncs before publishing the close snapshot, so a stale private cursor is not written back over Realm authority.
- Generic opened-container clicks now perform the same cursor resync before handling chest/container clicks.
- Quick-move is blocked while the cursor is non-empty, because sending shift-click through the current legacy quick-move path while cursor state is unsettled can poison later clicks.
- The ViaProxy inventory patch marker is now `v0.3.56-authoritative-cursor-resync`.

## What to test next

Use a Java client through `.\run-checked-bridge-latest.ps1 -RealmName "Example Realm" -SkipChecks`.

First pass should focus only on player inventory:

1. Open inventory.
2. Left-click pick up a stack.
3. Left-click place it into a different empty slot.
4. Close inventory, wait a few seconds, reopen inventory.
5. Try picking up and placing that same stack again.
6. Try one shift-click after a successful normal move.

If this patch is doing its job, a shift-click with a non-empty cursor should no longer poison the cursor model. Crafting should wait until this basic cursor state is stable.
