# v0.3.34 Live 2x2 Crafting Recipe DB

## Goal

Do not hard-code only one or three player-inventory crafting recipes. The Java player 2x2 grid should use the live Bedrock recipe set that the Realm sends in `crafting_data`.

## Finding from the latest run

v0.3.33 loaded cleanly and fixed the previous `MOB_EQUIPMENT` descriptor crash. The log shows the patched jar active, the player reaching spawn, the player inventory click path working, and Java slot `0` entering the local 2x2 crafting-output path. It also shows v0.3.33 still relying on the old tiny hard-coded recipe logic by logging `crafted local 2x2 recipe=log_to_planks`.

The same run also shows `crafting_data` arriving before play, which is the right source of truth for the broader recipe set.

## What changed

### Node relay

Added `src/bridgeCraftingRecipes.js`.

When the relay receives the Bedrock `crafting_data` packet, it now extracts every shaped/shapeless recipe that can fit in the Java player 2x2 grid:

- shaped recipes with `width <= 2` and `height <= 2`;
- shapeless recipes with up to four required input items;
- `shulker_box`-encoded shapeless recipes with up to four required input items;
- item-tag ingredients are preserved instead of dropped.

The relay writes the generated database to both:

```text
viaproxy-run/bridge-crafting-recipes-2x2.json
bridge-crafting-recipes-2x2.json
```

The `viaproxy-run` copy is the one the ViaProxy JVM should load because ViaProxy runs with `viaproxy-run` as its working directory.

### Patched ViaBedrock classes

`InventoryContainer` now loads the generated recipe database and matches the Java player 2x2 grid against it before falling back to the tiny emergency recipes.

The dynamic matcher supports:

- shaped recipe offset matching, so a 1x1 or 1x2 recipe can be placed anywhere it fits in the 2x2 grid;
- shapeless matching with no extra unmatched occupied slots;
- Bedrock network IDs;
- metadata wildcard `32767`;
- item-tag matching for the tags observed in the live 2x2 recipe export: `minecraft:planks`, `minecraft:logs_that_burn`, `minecraft:coals`, `minecraft:egg`, and `minecraft:metal_nuggets`.

The patcher now injects 8 class entries:

```text
InventoryTracker.class
Container.class
InventoryContainer.class
InventoryContainer$ClickSlot.class
InventoryContainer$CraftRecipe.class
InventoryContainer$BridgeIngredient.class
InventoryContainer$BridgeRecipe.class
InventoryContainer$BridgeRecipeDatabase.class
```

## Known limitations

This is still a compatibility shim. It uses the existing local inventory/cursor model plus legacy Bedrock normal `inventory_transaction`. It does not yet generate modern Bedrock `item_stack_request` craft/take requests. If crafted outputs appear and then snap back after authoritative replay, recipe matching is fixed but upstream transaction correctness still needs the modern item-stack-request path.

Crafting table / `WORKBENCH` is intentionally not included in this patch. That needs a real workbench container implementation after the player 2x2 grid is stable.
