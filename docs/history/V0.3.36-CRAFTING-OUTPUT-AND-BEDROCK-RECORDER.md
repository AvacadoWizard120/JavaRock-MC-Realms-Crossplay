# v0.3.36 crafting output + native Bedrock packet recorder

## Why this patch exists

v0.3.35 fixed the recipe DB filter, but the player 2x2 crafting behavior was still wrong:

- clicking the Java crafting output did not feel like grabbing the result;
- the bridge was auto-placing the result into an inventory slot instead of putting it on the cursor;
- the input stack could stay in the Bedrock player-only UI crafting grid after closing/reopening inventory;
- the bridge still needs native Bedrock packet baselines to implement modern `item_stack_request` crafting instead of guessing from Java/ViaBedrock behavior.

The latest terminal output confirmed the patch loaded and the filtered recipe DB was active:

```text
[java-compat] ViaBedrock inventory patch active: v0.3.35-crafting-table-recipe-filter-and-station-db
[bedrock-relay] Exported 2166 live Bedrock crafting_table 2x2 recipe(s) for ViaBedrock
[bedrock-relay] Preserved 535 non-crafting-table station recipe(s) for future smelting/workstation support
```

It also showed the bad output behavior: output clicks repeatedly logged `craft_2x2_take_output` / `craft_2x2_quick_move` and deposited the result into `destSlot=9`, while the Java cursor remained empty.

## What changed

### Player 2x2 output now behaves like a cursor pickup

Ordinary left-click on Java output slot `0` now does this locally:

```text
output click
  -> consume matched 2x2 input slots locally
  -> put crafted result on the Java cursor
  -> remember the input before/after state as a pending craft

destination click
  -> send one direct legacy Bedrock normal inventory_transaction
  -> include UI crafting-input consumption
  -> include destination inventory slot update
  -> clear the pending craft
```

This should make the result visibly attach to the cursor instead of disappearing into an invisible or stale destination slot.

### Shift-click output still inserts into inventory

`QUICK_MOVE` on output still attempts a direct insert into player inventory. It uses the same live crafting-table recipe DB as v0.3.35.

### Closing own inventory returns 2x2 inputs

When ViaBedrock closes the Java player inventory container, the patched `InventoryTracker.markPendingClose(...)` calls:

```java
inventoryContainer.bridgeReturnCraftingGridToInventory("player_inventory_close_return_2x2_grid")
```

That moves non-empty Bedrock UI crafting slots `28..31` back into the player inventory before the Java view is refreshed.

### Native Bedrock packet recorder mode

A new command/script starts only a Bedrock relay and packet census. It does not start ViaProxy. Use this to connect Minecraft Bedrock Edition directly to the local relay and record baseline packets for actions like:

- open inventory;
- move items into 2x2 crafting grid;
- take crafting output;
- close inventory with items left in the grid;
- use a crafting table;
- place/break blocks.

Run:

```powershell
.\run-bedrock-packet-recorder-latest.ps1 -RealmName "Example Realm"
```

Then add/connect from Minecraft Bedrock Edition:

```text
Server Address: 127.0.0.1
Port: 19133
```

If Minecraft for Windows cannot connect to localhost, run once as Administrator:

```powershell
CheckNetIsolation LoopbackExempt -a -n=Microsoft.MinecraftUWP_8wekyb3d8bbwe
```

Recorder output is written under `packet-census/` with full packet samples enabled for short, focused sessions.

## Known limitation

This still sends legacy `inventory_transaction` for the final commit. If the Realm rejects that and replays old inventory state, the next step is to implement modern Bedrock `item_stack_request` craft/take/consume actions using the native Bedrock packet recorder output as the reference.
