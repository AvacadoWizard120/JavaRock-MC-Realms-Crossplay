# v0.3.32 - Player 2x2 crafting output fix

## Why this patch exists

v0.3.31 finally made ordinary player-inventory mouse movement usable by handling Java `QUICK_CRAFT` drag packets. The next blocker was crafting: logs showed the 2x2 player crafting input slots were being clicked and mapped into Bedrock player-only UI slots, but Java slot `0` was still logged as `java_crafting_output_unsupported_here` and ignored.

The same run also showed crafting-table opening reaches the Realm, but ViaBedrock logs `Tried to open unimplemented container: WORKBENCH`, so workbench UI support is still a separate container-open implementation issue.

## What changed

The ViaBedrock inventory patch now handles Java player inventory slot `0` as a conservative local 2x2 crafting output slot.

Supported first-pass recipes:

- one vanilla overworld/nether log/stem/hyphae/wood item -> matching planks x4
- two vertically stacked planks -> sticks x4
- four planks in the 2x2 grid -> crafting table x1

On craft output click, the patch:

1. Reads player-only UI crafting inputs from Bedrock slots `28..31`.
2. Calculates the Java output for slot `0`.
3. On output click, consumes one item from each recipe input.
4. Places the crafted result into the first stackable/empty player inventory slot.
5. Sends a legacy Bedrock normal inventory transaction containing the input consumption plus output placement.
6. Immediately replays Java inventory content so the frontend does not revert.

## Known limitations

This is intentionally not a full Bedrock `item_stack_request` crafting implementation yet. It is a narrow survival-first bridge for the player 2x2 grid. If the Realm later corrects the inventory after these legacy transactions, the next step is to generate proper modern Bedrock `item_stack_request` craft actions instead of legacy normal inventory transactions.

Crafting tables are still not fully implemented. The current failure is at ViaBedrock's `CONTAINER_OPEN` handling for Bedrock `WORKBENCH`; it cancels that container type before Java gets a workbench screen. The next patch should add a Workbench container path instead of trying to fake it as a chest.

## Building / block interaction notes

The building logs still show a separate issue: placement uses `player_action` plus legacy `inventory_transaction item_use/click_block`, while break attempts often emit rapid `start_break` / `abort_break` bursts inside `player_auth_input`. That explains the janky building feel. This patch leaves that path mostly untouched to avoid regressing combat, eating, movement, and inventory while crafting is isolated.
