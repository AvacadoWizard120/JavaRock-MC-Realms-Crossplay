# v0.3.57 authoritative stack sanitizer

This checkpoint targets the June 12 inventory trace where ordinary Java cursor moves were accepted by the Realm, but 2x2 craft output was rejected and the crafting grid stayed server-side after close/reopen.

## What the trace showed

- The Realm accepted normal `take` and `place` item stack requests.
- The two craft output requests were rejected with status `50`.
- Both rejected craft requests took the output into `cursor[0]#6269`, even though the cursor was empty before the craft. Native Bedrock uses destination stack id `0` for an empty cursor.
- A later native place request sent `hotbar[5]#6269`, while the latest accepted server response for that destination slot was `#6298`.
- The Java close handler could report `no_grid_items_to_return` even though the Realm still had `CraftingInputContainer[28]#6269`.

## Patch

- `src/nethernetBedrockRelay.js`
  - Forces legacy 2x2 craft-output-to-empty-cursor requests to use cursor destination stack id `0`.
  - Adds `bridgeSanitizedItemStackRequestParams` to correct outbound native item stack request destination ids from the relay's accepted server-state tracker.
  - Guards against accidentally rewriting a take destination to the same request id that is creating the cursor stack.

- `patches/viabedrock-inventory/InventoryContainer.java`
  - Adds a remembered 2x2 crafting grid mirror.
  - Uses the remembered grid as a close-time fallback when the HUD container is empty.
  - Updates the remembered grid from native cursor moves, direct fallback moves, and local craft consumption.

- `src/viaProxyInventoryPatch.js`
  - Bumps the Java patch id to `v0.3.57-authoritative-stack-sanitizer`.
