# v0.3.29 deferred pickup + terrain prewarm

This patch is based on the latest uploaded `bedrock-realm-bridge-mvp.zip` after the v0.3.28 generic container shim test.

## What the latest test proved

The v0.3.28 patch is loaded: startup verifies 4 patched ViaBedrock class entries and reports `v0.3.28-generic-container-click-shim`. Java player inventory clicks reach the patched player inventory handler, and opened chest clicks reach the patched generic `Container.handleClick(...)` path. The remaining problem is not click routing; it is transaction shape and frontend authority.

The log shows normal mouse pickup/place still sends legacy `inventory_transaction` packets with cursor/global inventory semantics. Modern Realms appear to replay authoritative inventory content quickly enough that the Java frontend still feels like the item never leaves the slot. Shift-click and number-key swaps are more likely to work because they can be represented as direct slot-to-slot/container actions without a cursor source.

The log also shows a spawn timing regression: the Java client can enter the world before the chunks directly below the player are fully usable locally, causing a brief fall-through before the client is corrected back upward.

## Inventory change

v0.3.29 changes ordinary pickup/place behavior:

```text
old:
  pickup click -> send legacy container + GlobalInventory cursor transaction immediately
  place click  -> send another legacy cursor transaction

new:
  pickup click -> local Java cursor only; remember source container/slot
  place click  -> send one direct container-to-container transaction
```

This targets the specific failure mode where Realms reject/replay legacy cursor transactions. The first pickup now logs:

```text
[BedrockRealmBridge] deferred player pickup source ...
```

The second/place click now logs:

```text
[BedrockRealmBridge] committed deferred carried item reason=pickup_direct_commit ...
```

The same deferred commit model is also wired into generic opened containers such as chests.

## Terrain prewarm change

v0.3.29 delays the local Java-facing `play_status.player_spawn` packet briefly after the Realm reports spawn. This gives the already-streaming chunks a short prewarm window before the Java client starts applying local gravity.

Default:

```powershell
$env:NETHERNET_RELAY_TERRAIN_SPAWN_DELAY_MS = "1200"
```

Set it to `0` to disable while comparing spawn behavior.

## Still not solved

This is not full modern Bedrock inventory parity yet. If pickup/place still snaps back, the next required step is generating real modern `item_stack_request` packets with correct stack network ids, source container ids, and request/response tracking.

Crafting output slot 0 remains intentionally unsupported until item stack request generation exists.
