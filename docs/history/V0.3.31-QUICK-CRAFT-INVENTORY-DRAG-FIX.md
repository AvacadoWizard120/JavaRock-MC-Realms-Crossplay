# v0.3.31 quick-craft inventory drag fix

This patch is based on the run where ordinary inventory movement worked while a chest/container screen was open, but felt broken in the standalone player inventory.

## Diagnosis

The v0.3.29/v0.3.30 inventory shim was no longer blind to clicks. The important difference was the input shape:

```text
Chest/open container:
  input=PICKUP
  container_pickup_local_deferred
  container_pickup_direct_commit

Standalone player inventory drag:
  input=QUICK_CRAFT
  javaSlot=-999 button=0
  javaSlot=<hovered slot> button=1
  javaSlot=-999 button=2
  ignored reason=unsupported_input
```

So the player-inventory failure was not a slot-map failure anymore. It was Java's click-drag path. When the user clicks an item, drags it to another inventory slot, and releases, modern Java emits a `QUICK_CRAFT` sequence rather than a second ordinary `PICKUP` click. The bridge had been ignoring that sequence.

## Changes

```text
- Bumps the ViaBedrock inventory patch id to v0.3.31-quick-craft-inventory-drag-fix.
- Keeps deferred pickup/direct commit behavior from v0.3.29.
- Keeps the pending-close guard intended by v0.3.30.
- Handles player-inventory QUICK_CRAFT start/add/end packets.
- Handles generic container QUICK_CRAFT start/add/end packets.
- Treats the common one-hovered-slot QUICK_CRAFT add packet as a direct commit.
- Acknowledges QUICK_CRAFT boundary packets so the Java UI does not get corrected back immediately.
- Ignores PICKUP_ALL safely with a frontend snapshot instead of returning false.
```

## Expected startup

```text
[java-compat] ViaBedrock inventory patch active: v0.3.31-quick-craft-inventory-drag-fix
```

If startup still says an older patch id, the patched jar was not rebuilt or the local files were not overwritten correctly.

## Expected player-inventory logs

A click-drag move in the standalone inventory should now look like this:

```text
player inventory handleClick entry ... input=PICKUP carriedEmpty=true
sent Java inventory snapshot reason=pickup_local_deferred ... carriedEmpty=false
player inventory handleClick entry ... input=QUICK_CRAFT button=1 carriedEmpty=false
committed deferred carried item reason=quick_craft_direct_commit ... cursorEmpty=true
sent Java inventory snapshot reason=quick_craft_direct_commit ... carriedEmpty=true
```

## Remaining inventory work

This is still a legacy transaction shim. It should make the Java frontend and basic Realm-side inventory state behave better, but the long-term implementation should replace legacy normal `inventory_transaction` with Bedrock's modern `item_stack_request`/`item_stack_response` flow.

Next targets:

```text
1. Confirm regular inventory click-drag sticks.
2. Confirm chest inventory still works.
3. Test 2x2 crafting input slots after regular movement is stable.
4. Implement crafting output/take-result behavior.
5. Implement furnace/crafting-table container translators.
6. Replace legacy inventory_transaction with item_stack_request for authoritative parity.
```
